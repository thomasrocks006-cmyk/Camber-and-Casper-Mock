Memory Types: How Context Persists
The Memory Layer stores 4 types of knowledge that both VICKI and Concierge access:

1. Episodic Memory — Time-Anchored Events
What: Specific events that happened at specific times
Retention: Permanent
Examples:

"Owner approved invoice #4521 for ABC Ltd on 2026-04-15"
"Changed brake supplier from XYZ to ABC Auto Parts on March 15 due to pricing"
"Lost deal with ConstructCo in September—timing wasn't right (but they just got funding)"
Used By:

VICKI: "Based on your decision last month to switch suppliers..."
Concierge: "Last time Michael visited, his car got a parking fine—I've validated parking this time"
2. Semantic Memory — Timeless Facts
What: General knowledge about the business, preferences, and relationships
Retention: Permanent
Examples:

"Owner prefers audio briefings over written reports"
"ABC Ltd always pays 45 days late but they always pay"
"Never compete on price—quality and service first"
"Owner wants to work only 3 days/week by 2027"
Used By:

VICKI: "I'll give you the 3-minute audio brief instead of the written report"
Concierge: "Based on your goal of 3-day work weeks, I've scheduled Fridays as no-meeting days"
3. Working Memory — Current Context
What: What's happening RIGHT NOW in this conversation/task
Retention: 4 hours (then archived or promoted to episodic)
Examples:

"Currently discussing cash flow for Q2"
"Owner asked about competitor pricing 3 messages ago"
"Meeting prep in progress for 2pm investor pitch"
Used By:

VICKI: Maintains conversation thread without repetition
Concierge: Coordinates multi-step tasks (email draft → approval → send)
4. Relationship Memory — People & Entities
What: Everything known about customers, suppliers, competitors, team members
Retention: Permanent (updated continuously)
Examples:

"Michael Chen (ConstructCo CEO) - High-value prospect, just raised $2.4M"
"Sarah (Sales Lead) - High-I personality, prefers warm personal communication"
"Workshop Wizard (competitor) - Dropped prices 15% on April 21, threat level: medium"
Used By:

VICKI: "Based on Michael's funding round, this is the perfect time to pitch"
Concierge: "Sarah prefers SMS over email, sending confirmation via text"

Technical Implementation: How It's Built
Memory Layer Architecture (pgvector)
# Memory Service — The Shared Brain

class MemoryService:
    async def store(
        entity_id: str,
        user_id: str,
        memory_type: MemoryType,  # EPISODIC | SEMANTIC | WORKING | RELATIONSHIP
        content: str,
        importance: float,  # 0.0 - 1.0
        metadata: dict = None,
        agent_run_id: str = None
    ):
        """
        Store a memory that ANY system can later retrieve.
        
        Examples:
        - VICKI stores: "Owner asked about cash on 2026-04-22"
        - Concierge stores: "Michael Chen prefers 2pm meetings"
        - Archivist stores: "Meeting notes from investor pitch"
        """
        
    async def query(
        entity_id: str,
        user_id: str,
        query: str,
        memory_types: list[MemoryType] = None,
        limit: int = 5,
        min_importance: float = 0.0
    ) -> list[Memory]:
        """
        Retrieve relevant memories for ANY system to use.
        
        Uses:
        - pgvector semantic similarity search
        - Recency weighting
        - Importance scoring
        - Access count tracking (prevents decay)
        
        Examples:
        - VICKI queries: "What hiring decisions were made?"
        - Concierge queries: "What's owner's meeting preference?"
        - Silicon CFO queries: "What's the cash safety minimum?"
        """
Cross-System Integration Pattern
# How VICKI and Concierge share context

class VICKIEngine:
    async def chat(self, message: str, entity_id: str, user_id: str):
        # 1. Query shared memory
        memory_context = await self.memory_service.query(
            entity_id=entity_id,
            query=message,
            limit=5
        )
        
        # 2. Pull fresh data from pillars
        cash_data = await self.silicon_cfo.get_cash_position(entity_id)
        invoices = await self.invoice_repo.get_overdue(entity_id)
        
        # 3. Synthesize response
        response = self._generate_response(message, memory_context, cash_data)
        
        # 4. Store NEW memory (for Concierge to use later)
        await self.memory_service.store(
            entity_id=entity_id,
            user_id=user_id,
            memory_type="episodic",
            content=f"Owner asked: {message}. VICKI discussed: {response[:200]}",
            importance=0.6
        )
        
        return response


class ConciergeEngine:
    async def draft_email(self, recipient: str, entity_id: str):
        # 1. Query SAME shared memory that VICKI uses
        context = await self.memory_service.query(
            entity_id=entity_id,
            query=f"What do I know about {recipient}?",
            memory_types=["relationship", "episodic"],
            limit=10
        )
        
        # 2. Access Scout/Ghost/War Room data (same pillars VICKI uses)
        contact_profile = await self.scout.get_profile(recipient)
        competitive_context = await self.war_room.get_intel(recipient)
        
        # 3. Draft with full context
        draft = self._generate_personalized_email(
            recipient, 
            context,
            contact_profile,
            competitive_context
        )
        
        # 4. Store memory (for VICKI to reference later)
        await self.memory_service.store(
            entity_id=entity_id,
            memory_type="episodic",
            content=f"Concierge drafted email to {recipient} re: {draft.subject}",
            importance=0.4
        )
        
        return draft
Key Pattern: Both engines READ and WRITE to the same Memory Layer. This creates automatic context sharing without manual integration.
